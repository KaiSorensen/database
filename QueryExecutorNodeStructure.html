Query Executor Node Structure

This page summarizes the current query-executor parse tree as it exists in the codebase. The tree itself is the executor. Each node validates its structure, delegates tabular shape logic into Data, and then either returns read results or commits mutations through the CRUD layer.

Reading Guide

The metadata branches stay simple. The data branch is where selection, row-wise operations, column-wise operations, joins, and CRUD validation live. Update is intentionally split into two semantic roles: choosing target rows and computing replacement values.

Top-Level Structure

Metadata

Object Query

Object CRUD nodes manage object metadata. They do not use the tabular data branch.

Object

ObjectCreateNode / ObjectReadNode / ObjectUpdateNode / ObjectDeleteNode

These stay close to the CRUD engine’s schema-level operations.

Metadata

Attribute Query

Attribute CRUD nodes manage attribute metadata, including adding or removing stored columns.

Attribute

AttributeCreateNode / AttributeReadNode / AttributeUpdateNode / AttributeDeleteNode

These operate on object-attribute definitions rather than on row data.

Data

Data Query

Data CRUD nodes own the executable subtree that works with selections, operations, joins, and mutation validation.

CRUD Roots

DataReadNode / DataCreateNode / DataUpdateNode / DataDeleteNode

These are the roots that decide whether the subtree is used for reading, creating, updating, or deleting.

Data Branch

Selection

SelectionNode projects columns and applies filter expressions. It is the most common leaf-like node in the data branch, even though it still transforms the current Data view.

  • Input: current Data context
  • Output: filtered/projected Data
  • Validation: selected columns exist, filter columns exist

Row Operation

RowOperationNode computes one derived column. It can consume one child result or merge two child results when their row bindings match exactly.

  • Unary or binary input
  • Produces one derived column
  • Exact row-set match is required for binary non-join paths

Column Operation

ColumnOperationNode reduces one column to one scalar. The scalar is still represented as a one-row, one-column Data object.

  • Consumes a single-column input
  • Produces a scalar-shaped Data
  • Can also merge two child results before reduction when row bindings match

Join

JoinNode makes row-shape reconciliation explicit. It is the escape hatch when two branches do not already share the same row bindings.

  • Current join mode: INNER
  • Current key shape: one key column per side
  • Output rows carry bindings from both joined objects

CRUD Root Shapes

DataRead

One active child path among selection, row operation, column operation, or join. The result only needs to be readable.

DataCreate

One active child path among selection or row operation. A selection-shaped result can become full rows. A row-operation result can become a new stored column.

DataDelete

One active child path among selection or row operation. The result must validate as a full row set or a full column set before deletion begins.

DataUpdate

This root is different. It has two semantic branches: one branch chooses the stored rows to update, and another branch computes the replacement values. 

DataUpdate
├── targetSelectionChild
└── valueExpressionChild
    ├── valueSelectionChild
    ├── valueRowOperationChild
    ├── valueColumnOperationChild
    └── valueJoinChild

Update Rules

Target Selection

The target branch must identify real stored rows for the object being updated. This is where row identity comes from.

  • The target result must still bind to stored row indexes
  • The update does not infer targets from the value branch
  • If the target rows are ambiguous, the update fails

Scalar Broadcast

If the value branch returns a scalar, that scalar may update one target attribute across every selected target row.

  • Value shape must be exactly one row and one column
  • Exactly one target attribute is allowed
  • The same value is written to every targeted row

Per-Row Replacement

If the value branch is not scalar, it must fit the target shape exactly.

  • Exact row-binding match with the target selection
  • One produced column per target attribute
  • No write begins unless the fit check succeeds

Safe Failure

The design intentionally avoids rollback code. Instead, update requests are staged and type-checked before the first write.

  • Shape fit is validated first
  • Target attribute types are checked before commit
  • If any check fails, nothing is written

Current Testing Coverage

In-Memory Executor Tests

The executor is tested directly against in-memory Data fixtures for filtering, operations, joins, and tree execution.

Persisted Integration Tests

Query-executor integration tests seed data through CrudEngine, load it with Data.fromCrudEngine(...), execute a root tree, and assert on the real persisted database state.

Mutation Coverage

Current integration coverage includes row create, derived-column create, row-aligned update, scalar broadcast update, row delete, and column delete.

Wrapper Readiness

A future query-executor wrapper can simplify orchestration, but the parse tree and the runtime data model are already testable as they stand today.
